Presentación

Moca platform es una startup que ofrece una herramienta de marketing basada en la localización. Se trata de un modelo B2B, donde el partner integra un SDK en su app que alimenta un dashboard con analíticas, mapas y herramientas para lanzar campañas (notificaciones).

Actuales roles en MOCA

CEO

Meetings con potenciales partners, conferencias, etc.

CTO

50% tareas que la CEO. El otro 50% dividido entre CTO real y Product Owner

Responsable de Marketing

No se la tiene muy en cuenta en la creación del producto. Gestiona la imagen que se proyecta de MOCA en la landing. Ayuda a los clientes a usar la herramienta.

Ventas

Busca partners y acuerdos con posibles clientes.

Dissenyadora gráfica

Front End developer

Backend developer

SDK Developer

Client support

Workflow Actual

Analizamos la manera en que actualmente funciona MOCA a la hora de desarrollar y mejorar el producto.

1. Roadmap

Suele existir un Roadmap de lo que se quiere hacer a lo largo del año.

Pero se salta a las primeras de cambio...

2. CTO Inception

El CTO tiene una idea o para cerrar un trato con un nuevo partner llega al acuerdo de crear una nueva funcionalidad.

El CTO suele acumular varias funcionalidades nuevas y cuando tiene entre 1-4 suele comunicarse al grupo de dev convocando un meeting (support también está en este meeting).

3. Meeting

  • Dura 1 hora o más
  • Se describe lo que se tiene que hacer a un nivel bastante técnico en una pizarra
  • A veces se estima (muy por encima) y se reparten los features en una línea temporal (otras no)
  • Esta pizarra está visible para todos y no se borra

Siempre se estima de forma muy optimista. El CTO a veces impone unos tiempos muy ajustados e irreales

4. JIRA

  • El equipo de DEV suele usar la herramienta JIRA
  • El CTO no participa en crear issues ni mantener un backlog
  • No se usa la parte de sprint ya que el tiempo es custom para cada grupo de features

5. CTO Interruption...

Muchas veces el CTO viene con algo más importante y interrumpe el schedule

6. Desarollo

Cuando un developer ha terminado su desarrollo, se sube normalmente en un entorno de pruebas y support hace un testing manual

7. Producción

Cuando todo está OK se sube a producción. Si la tarea tiene una representación en Jira se cierra.
