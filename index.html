Workflow Actual

Cómo funciona MOCA a la hora de desarrollar y mejorar su producto

1. Roadmap

Suele existir un Roadmap de lo que se quiere hacer a lo largo del año.

Pero se salta a las primeras de cambio...

2. CTO Inception

El CTO tiene una idea o para cerrar un trato con un nuevo partner llega al acuerdo de crear una nueva funcionalidad.

3. Meeting

  • Dura 1 hora o más
  • Se describe lo que se tiene que hacer a un nivel bastante técnico en una pizarra
  • A veces se estima (muy por encima) y se reparten los features en una línea temporal
  • Esta pizarra está visible para todos y no se borra

4. JIRA

  • El team de DEV suele usar la herramienta JIRA
  • El CTO no participa en crear issues ni mantener un backlog
  • No se usa la parte de sprint

5. CTO Interruption...

Muchas veces el CTO viene con algo más importante y interrumpe el schedule

6. Desarrollo

Cuando un developer ha terminado su desarrollo, se sube normalmente en un entorno de pruebas y support hace un testing manual

7. Producción

Cuando todo está OK se sube a producción. Si la tarea tiene una representación en Jira se cierra.

Ejemplo: TEAMS

Ejemplo de nueva funcionalidad en el DASHBOARD.

Se trata de poder crear TEAMS, con dos nuevos roles y us permisos: ADMIN y USERS.

El PO mantiene el BACKLOG

  • EPIC-001: Como usuario quiero poder crear TEAMS y asignar roles con diferentes niveles de acceso
    • US-001: Como usuario quiero poder crear un TEAM
    • US-002: Como usuario quiero poder registrar a un usuario ADMIN a un team determinado
    • US-003: Como ADMIN quiero poder crear un usuario tipo USER
    • US-004: Como ADMIN quiero poder añadir y eliminar acciones a un determinado USUARIO
    • US-005: Como USER/ADMIN quiero poder hacer login al team
    • US-006: Como USER/ADMIN quiero poder ver mi team dependiendo de las acciones permitidas por el ADMIN

El equipo participa en el SPRINT PLANNING

  • Dirigida por el Scrum Master
  • Dev team + PO + QA
  • Debate técnico, subtareas, puntuación

Finalmente el Product Owner elige las tareas que pasan al sprint, teniendo en cuenta que la capacidad del equipo es de 50 story points.

La prioridad en este ejemplo, marcada por el Product Owner, viene dada por el orden de la US en la lista.

Dailies

  • La organiza el Scrum Master
  • Se discute el estado de las tareas
  • SM gestiona bloqueos
  • Reuniones de pie
  • Cada uno expone bloqueos, TODO and DONE
  • Debe ser breve

DEMO

  • Supone todas las tareas en DONE
  • Se entrega Release -> CI -> DEMO
  • SM organiza DEMO para todos
  • Ronda preguntas
  • PO valida el trabajo

RETRO

  • Reunion que organiza el Scrum Master
  • Procedimientos a seguir haciendo
  • Procedimientos a cambiar
  • Tareas para la próxima iteración
